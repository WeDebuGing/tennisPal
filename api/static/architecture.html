Request Flow

sequenceDiagram
    participant B as Browser
    participant N as Nginx :80
    participant F as Flask :5001
    participant D as SQLite

    B->>N: GET / (or any route)
    N->>F: proxy_pass
    F->>F: Serve built frontend (static/)
    F-->>B: HTML + JS bundle

    Note over B: User interacts with SPA

    B->>N: POST /api/auth/login
    N->>F: proxy_pass
    F->>D: Query user by email
    D-->>F: User record
    F->>F: Verify password, create JWT
    F-->>B: { token: "eyJ..." }

    Note over B: Subsequent requests include JWT

    B->>N: GET /api/posts (Authorization: Bearer ...)
    N->>F: proxy_pass
    F->>F: Validate JWT
    F->>D: Query posts
    D-->>F: Post records
    F-->>B: { posts: [...] }

Match Lifecycle

stateDiagram-v2
    [*] --> PostCreated: User creates "Looking to Play"
    PostCreated --> InviteSent: Another user sends invite
    InviteSent --> InviteAccepted: Recipient accepts
    InviteSent --> InviteDeclined: Recipient declines
    InviteDeclined --> PostCreated: Post remains active
    InviteAccepted --> MatchScheduled: Match created
    MatchScheduled --> ScoreSubmitted: One player submits score
    ScoreSubmitted --> ScoreConfirmed: Other player confirms
    ScoreSubmitted --> ScoreDisputed: Other player disputes
    ScoreConfirmed --> Completed: Elo updated
    MatchScheduled --> Cancelled: Either player cancels
    Completed --> Reviewed: Players tag each other
    Completed --> [*]
    Reviewed --> [*]

Key Design Decisions

DecisionRationale
SQLite over PostgresSimple deployment, single-server, <1000 users. Upgrade path: swap DB URI
Elo rating systemFair, well-understood ranking. Default 1200, K-factor adjustable
Request → Accept flowPrevents instant match-claiming; gives both players agency
Score confirmationBoth players must agree on score for Elo to update
Review tags (not free text)Structured feedback, aggregatable, less toxic than open reviews
JWT (not sessions)Stateless auth, mobile-friendly, no server-side session store
